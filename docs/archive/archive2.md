# 前一阶段工作总结

本阶段的毕业设计工作围绕“面向异构存储架构的高并发缓存系统设计与实现”这一题目展开，聚焦于现代存储环境下的页面缓存管理、并发控制、异步 I/O 组织方式、页面替换策略以及系统可观察性等核心问题。本项目并不是单纯实现一个“能跑的 Buffer Pool”，而是从设计阶段把“高并发数据面”和“低侵入观测面”同时纳入系统边界，力图构建一个既能运行、又能被分析和解释的缓存系统实验平台。和传统课程设计中“先搭一个能运行的简化版系统”不同，我在本阶段更强调系统边界的清晰性、模块解耦的合理性、并发语义的正确性以及后续实验与展示的可支撑性。围绕这一思路，前一阶段的工作主要体现在文献调研与方案修正、核心引擎实现、I/O 与写回路径建设、可观察性接口设计、测试与持续集成体系建设以及 Web 端实验展示支撑平台建设等方面。

第一，在文献调研与研究路线修正方面，重点学习和梳理了数据库系统中 Buffer Pool 的经典机制、现代存储环境下异步 I/O 的组织方式，以及系统可观察性的工程实践。具体包括：围绕基于页的存储管理模型分析缓冲池在内存与外存之间的角色；围绕 LRU、Clock、LRU-K 等经典替换算法理解缓存命中率与工作负载分布之间的关系；围绕现代 NVMe SSD 与 io_uring 的研究资料理解高并发随机 I/O 场景下传统同步写回路径的局限；围绕 LeanStore、InnoDB、Bustub 等系统的架构分析工业级系统与教学系统之间的差异；围绕近年来对数据库配置失配、异构存储适配与性能归因的研究，理解“仅有吞吐量数字而缺乏运行时解释能力”并不足以支撑系统研究。经过这部分调研后，逐步明确：本科阶段不宜将系统边界铺得过大，如果同时追求完整 GUI、异构延迟模拟、自动调优、共享内存监控平面和底层缓存引擎全部完成，容易造成任务发散、每一项都停留在浅层原型。因此，本阶段“先做实高并发缓冲池核心和可观测接口底座，再在其上叠加监控展示与实验支撑”，使项目从一开始就朝着“可运行、可验证、可扩展”的研究型工程方向推进。

第二，在系统核心架构设计方面，本阶段已搭建起 TelePath 项目的整体工程架构，并确立了以“数据面与观测面解耦”为设计核心的实现框架，如图1所示。项目采用现代 C++17 标准与 CMake 组织，通过对 `BufferManager`、`BufferHandle`、`BufferDescriptor`、`DiskBackend`、`Replacer` 以及 `TelemetrySink` 等关键模块的深度抽象，建立了清晰的职责边界。具体而言，`BufferManager` 统筹缓冲池生命周期与全局状态维护，其核心接口如图2所示；`BufferHandle` 通过 RAII 机制确保页面句柄在作用域内自动完成 Pin/Unpin 语义；`BufferDescriptor` 精确刻画单帧的物理与逻辑状态；`DiskBackend` 封装了底层异步 I/O 的读写细节；`Replacer` 及其衍生实现负责页面淘汰策略的动态决策；而 `TelemetrySink` 则承担了观测平面的事件捕获与快照导出职责。这种模块化的分层设计有效避免了并发控制、磁盘 I/O 与监控逻辑的过度耦合，为后续引入异构存储后端、多策略性能对比以及可视化监控平台的接入提供了良好的扩展性与灵活性。

第三，在缓冲池核心能力实现方面，本阶段已经不再停留在“仅能读写页面”的简单原型，而是基本具备了一个可实验的并发缓冲池内核应有的关键语义。当前实现已经支持页面读入、页面驻留检查、页句柄获取、显式释放句柄、脏页标记、单页刷盘、全量刷盘以及运行时快照导出等基础能力。在并发控制上，已支持同一页面被多个线程同时请求时的缺页协调机制，使得真正执行缺页安装工作的只有一个线程，其余线程等待安装完成后复用结果，避免重复装载和状态竞争；在驻留页访问上，已具备面向 resident page 的快速读取路径；在淘汰与替换上，已能配合替换器和页表对空闲帧、可淘汰帧和脏页帧进行区分处理；在快照导出上，已能输出帧级别状态信息，为后续前端展示和实验结果解释提供依据。与只强调功能能跑起来不同，在这一阶段更重视缓冲池内部状态机的一致性与异常分支处理，例如句柄失效、资源耗尽、缺页失败恢复、等待与淘汰交错等情况都已在实现和测试中得到考虑，例如，缺页处理逻辑如图3所示。

第四，在替换策略与并发语义探索方面，本阶段已经完成了 LRU、LRU-K 和 Clock 三类替换器的实现，并将其真正接入缓冲池运行路径，如图4所示。替换策略的接入为后续实验带来了两个价值：一方面，可以在不同负载下比较不同策略的行为差异，为后续论文实验部分积累基础；另一方面，也迫使缓冲池核心在 frame 可淘汰状态、pin count 变化、缺页安装和写回冲突等问题上实现更加清晰的状态转换逻辑。相比只实现单一替换算法，本阶段完成多策略并行支持更能体现系统设计的实验平台属性。此外，还重点处理了并发访问下若干容易被忽视但会影响正确性的语义问题，例如同页竞争、失效句柄检查、淘汰后恢复、刷盘期间再次置脏、等待者与拥有者之间的状态传递等。这些工作虽然不像界面功能那样容易展示，但它们实际上构成了整个项目最核心、最有技术含量的部分。

第五，在磁盘后端抽象与异步式 I/O 路径建设方面，本阶段已经完成统一的 `DiskBackend` 抽象，并在此基础上实现了稳定的 POSIX fallback 路径和面向支持环境的 `io_uring` 后端路径。io_uring是Linux内核于2019年引入的一种高性能异步I/O接口，旨在通过共享内存环形队列减少系统调用和内存拷贝，提升I/O性能，其工作结构如图5所示。重点是围绕统一的提交/完成模型组织底层 I/O，使上层缓冲池可以通过相同抽象面对不同后端。当前后端实现已经考虑到读写请求标识、完成事件回传、后端能力协商以及 fallback 与 native path 的分离，使得系统既能在普通环境下稳定运行，也为后续更深入的原生异步 I/O 优化留下空间。需要说明的是，在当前阶段，`io_uring` 只作为正确性优先的实验路径，而不是宣称极限性能，这样可以保证系统先建立起正确、可调试、可验证的异步接口边界，避免在基础语义尚未稳固时就过早追求性能数字。

第六，在异步写回路径与后台管理子系统的构建上，本阶段攻克了项目中技术风险最高且实现逻辑最复杂的模块。具体而言，系统引入了集中式完成事件分发机制，通过统一的调度平面精确管理异步 I/O 事件流，有效规避了多线程环境下磁盘完成事件被无序消费的竞争风险。在调度策略上，实现了前/后台刷盘任务队列的分离与流量控制，通过配置批量提交上限建立了合理的优先级调度模型，在保障前台显式 Flush 请求响应速度的同时，有效防止了后台预清理任务对系统资源的长期挤占，其流程如图6所示。此外，项目实现了基于脏页高低水位的自适应 Cleaner 机制，并针对“写回过程中页面再次置脏”等复杂一致性场景，设计了严密的脏位确认逻辑，从根本上杜绝了脏状态误报导致的数据丢失风险，其流程如图7所示。这部分工作通过对并发同步、任务排队与故障传播路径的精细建模，使 TelePath 具备了处理生产级 I/O 负载的鲁棒性，是系统从功能原型走向高可靠存储引擎的关键跨越。

# 下一阶段工作计划

下一阶段的工作将围绕“继续增强系统可观察性、补充更系统的实验结果、完善论文与答辩材料”三条主线推进。总体策略不是盲目再开很多新方向，而是在当前核心引擎已经具备较好雏形的基础上，继续把已有成果做深、做实、做成可直接用于论文和答辩展示的完整材料。需要明确的是，下一阶段的推进重点不只是继续完善缓存系统本身，更要把 TelePath 的可观察性优势真正转化为可展示、可分析、可写入论文的核心成果。换言之，下一阶段的重点将从“以实现为主”逐渐转向“实现、实验、总结和展示并重”。

第一，继续增强 TelePath 的观测能力与结果表达能力。当前系统已经具备 `TelemetrySink`、快照接口、基础计数器和 Web 控制台等支撑，但从毕业设计论文的完整度来看，仍有较大提升空间。下一阶段计划进一步扩充观测事件的种类和记录粒度，使其不仅能记录命中、未命中和读写次数，还能更清晰地刻画淘汰、刷盘、后台 cleaner 活动以及队列状态变化；在不影响核心数据面稳定性的前提下，继续探索更轻量的状态导出机制，例如共享内存、Socket 或其他 IPC 形式的适配方案，并至少形成一套能够自洽说明的设计方案和原型验证结果；结合现有 Web 控制台，对运行结果展示形式做进一步整理，使系统行为、实验参数与结果输出之间的对应关系更加直观。即使共享内存完整方案在本科阶段不一定全部做完，也应确保“可观察性优先”的架构思想在论文中有清晰证据支撑，而不是停留在概念描述层面。

第二，围绕系统实验与 benchmark 数据继续展开更系统的整理和补充。当前项目虽然已经具备 benchmark 程序、线程 sweep 脚本和 GitHub Actions benchmark 工作流，但实验结果尚需从“能跑”进一步提升到“能分析、能比较、能支撑结论”。下一阶段计划重点补充以下几类实验：不同线程数下系统吞吐量和命中率变化；不同替换策略在热点负载、均匀负载和顺序共享/顺序分离负载下的表现差异；不同缓冲池规模与 block 数配置对系统行为的影响；显式刷盘、后台 cleaner 与脏页积压之间的关系；以及 fallback 路径和 native `io_uring` 路径在可验证环境下的行为差异。对于 WSL2 本地环境不适合直接完成的部分原生实验，我将继续利用 GitHub Actions 的 Linux runner 进行自动化验证并保留 artifact、日志和 CSV 结果，以保证实验流程具备可复现性和客观性。通过这部分工作，希望能够在中后期论文中形成若干真正有分析价值的曲线图、统计表和结果对比图，而不是仅有零散截图。

第三，继续补强工程化材料与工作量证明链条。由于本课题的一个重要优势在于其不仅实现了底层系统，还建设了较完整的测试、脚本、CI 和 Web 支撑体系，因此下一阶段计划对这些材料进行更系统的归档与梳理。例如，整理测试子系统分布、测试类别说明、CI 工作流页面、benchmark workflow 结果、项目代码结构与演进阶段截图等，使这些内容在论文和答辩展示中不再只是“辅助材料”，而成为反映工程工作量、开发规范性和系统完整度的重要证据。同时，还会继续完善 build/test/ benchmark/web 相关脚本和文档，保证项目在本地和 CI 环境中的行为尽可能一致，减少“只在某一台机器能跑”的问题，使系统具备更高的可复现实验属性。

第四，逐步完成毕业论文正文、插图和答辩展示材料的撰写。下一阶段将以现有代码实现和实验结果为基础，系统整理课题背景、国内外研究现状、总体架构设计、关键模块实现、测试与实验方法、结果分析、存在问题和后续工作等内容，并将本阶段技术路线调整的原因说明清楚：即在保证课题主线不变的前提下，优先完成最有技术含量、最能支撑毕业设计质量的核心引擎部分。对于尚未完全完成的内容，如更成熟的共享内存事件导出、更深入的 `io_uring` 性能优化、以及更丰富的实时可视化表达。总体而言，下一阶段的目标，是把当前已经较为扎实的工程成果进一步沉淀成结构完整、论证充分、图表翔实的毕业设计论文和答辩材料，使项目既能体现系统实现深度，也能体现研究与工程结合的完整性。

# 存在问题

当前项目推进中存在的问题，主要不是选题方向或系统主线错误，而是一个底层系统工程在有限时间、有限实验环境下需要持续处理的实现边界与验证边界。首先，WSL2 环境在原生 `io_uring` 验证方面存在天然限制，这意味着本地环境更适合作为开发、调试与普通回归测试平台，而不完全适合作为所有 native 异步 I/O 行为和性能结论的唯一依据，而是通过引入 GitHub Actions 中的原生 Linux runner，将 `io_uring` 相关测试和 benchmark 验证纳入持续集成流程，用“本地开发 + 远端验证”的方式补足环境不足。这一问题虽然客观存在，但也推动项目形成了更规范的跨环境验证机制。

其次，可观察性虽然已经在架构层面完成接口设计并具有初步实现，但共享内存事件流、独立监控进程和更丰富的动态展示能力尚未完全落地。究其原因，并非项目忽视监控面，而是因为我在阶段推进中有意识地优先保障数据面核心的正确性和稳定性，先完成缓冲池并发语义、写回调度、后端抽象和测试体系，再向外扩展监控平面。如果在前期过早把大量时间投入到 GUI 和展示层，反而可能导致最关键的缓存引擎部分深度不足。下一阶段将进一步补足这一部分，使观测能力从接口预留进一步走向更直观的运行时表达。

再次，当前虽然已经具备较丰富的测试、benchmark、CI 和 Web 展示支撑，但从论文与答辩需要的角度看，实验图表和结果分析仍需继续充实。底层系统项目往往存在一个特点，即代码实现工作量很大，但如果没有经过进一步整理，评审教师未必能直观看到这些工作背后的技术深度和系统复杂度。因此，下一阶段除了继续完善实现外，还需要把已有测试结果、benchmark 结果、CI 工作流记录、Web 控制台展示结果以及代码结构演进过程更有条理地整理成图表和材料，使“工作量大”“模块多”“验证充分”这些事实能够更直接地体现出来。总体来说，当前存在的问题是可以继续推进和逐步化解的工程问题，并不会动摇本课题已经取得的中期成果。

# 其它

本项目采用现代化开源软件标准进行开发与托管，源码与实时测试状态可通过 git 查阅。在安装了git的环境下，可通过git命令获得源代码：`git clone https://github.com/adj-matrix/TelePath.git`

仓库所有者即为毕设作者本人。系统的 CI 持续集成流、性能基准测试报告均可在该仓库的 Actions 页面公开审核。

因本课题兼具长期研究价值，结题后可能会持续演进更新。毕业设计评审请以结题时提交的留档源码压缩包为准。
